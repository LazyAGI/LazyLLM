# 一、本周使用 Cursor 完成的主要任务（按事项列出）

## 任务 1：LazyLLM Tracing 机制（Phase 1）全量开发与合并

### 1. 任务背景

* **要解决的问题：** LazyLLM 框架缺少统一的 Tracing / 可观测性机制。用户在使用 Module、Flow 等组件进行 RAG、多轮对话等场景时，无法方便地追踪调用链路、定位性能瓶颈或排查错误。需要从零搭建一套基于 OpenTelemetry 的 Tracing 系统，并对接 Langfuse 后端，同时在 Module / Flow 层面自动埋点。
* **为什么用 Cursor：** 本次涉及 19 个文件、1200+ 行新增代码，包括 tracing runtime 核心、后端抽象层、hook 注入机制、module/flow 埋点改造等多个模块。使用 Cursor 可以加速 boilerplate 代码生成、辅助设计 API 接口、快速迭代架构方案。

### 2. 使用 Cursor 的过程

**阶段一：Tracing Runtime 核心 + Langfuse 后端**

* **初始 Prompt（大意）：**
  > "我要为 LazyLLM 框架实现一个 tracing 系统。基于 OpenTelemetry SDK，支持 Langfuse 后端。需要：1）TracerProvider 初始化与延迟加载；2）start_span / finish_span 生命周期管理；3）trace context 在 globals 中传播；4）Langfuse 属性映射（session_id、user_id、tags）。请生成 `lazyllm/tracing/runtime.py` 的核心实现。"

* **第一次输出效果：** 基本框架能用（约 70% 可用），但存在以下问题：
  - 将 TracerProvider 放在模块级别直接初始化，没有做延迟加载（lazy init），导致 import 时即触发初始化
  - trace_context 的 normalize 逻辑缺失，`request_tags` 等字段没有做类型保护
  - Langfuse 属性名使用了过时的命名约定

* **引导与调整过程：**
  1. **拆分问题：** 将"一次性生成整个 tracing 模块"拆为：runtime 核心 → backends 抽象 → Langfuse 实现 → hook 注入，每步单独迭代
  2. **补充上下文：** 将 `lazyllm/common/globals.py` 和 `lazyllm/configs.py` 贴给 Cursor，让它理解 globals 传播机制和 config 体系
  3. **限制输出格式：** 要求 Cursor "只输出类和函数定义，不要加 main 测试代码"
  4. **纠正架构问题 Prompt 调整：** "TracerProvider 必须延迟初始化，在首次 start_span 时通过 double-checked locking 完成初始化。使用 threading.RLock 保护。"

* **最终效果：** 经过约 3-4 轮迭代，runtime.py 核心代码达到可 review 的质量。

**阶段二：Hook Provider Registry 解耦**

* **背景：** 初始版本中 `module.py` 和 `flow.py` 直接 import `tracing.hook`，耦合度高。
* **Prompt：**
  > "当前 module.py 和 flow.py 直接 from ..tracing.hook import resolve_tracing_hooks，这导致 tracing 和 module/flow 之间形成了强耦合。请设计一个 hook provider registry 机制：在 hook.py 中增加 register_builtin_hook_provider 和 resolve_builtin_hooks，tracing 模块通过注册自身的 provider 来解耦。"

* **Cursor 输出：** 一次生成的 `register_builtin_hook_provider` + `resolve_builtin_hooks` 代码基本可用，只需微调变量命名。

**阶段三：Callable Span Support + Semantic Type + Usage Collection**

* 这三个功能在 Phase 1 合并后的增量开发中完成（目前在开发分支上）。
* **使用 Cursor 的方式：** 将已有的 `runtime.py`、`flow.py` 等文件作为上下文，逐步 Prompt：
  - "在 flow.py 的 `_run_callable` 中为非 Flow/Module 的 callable 对象添加 span 追踪"
  - "在 runtime.py 的 `_base_attributes` 中增加 semantic_type 属性采集，支持从 target.__semantic_type__ 获取"
  - "添加 set_usage 方法收集 LLM 的 token usage（prompt_tokens, completion_tokens）"

**阶段四：跨线程 Trace Context 重建（当前进行中）**

* **问题：** LazyLLM 的 Parallel Flow 使用线程池并发执行子任务，新线程中 OpenTelemetry Context 丢失，导致子 span 无法挂载到父 span 下。
* **Prompt：**
  > "在 start_span 中，当检测到当前是 root span 但 trace_ctx 中携带了 trace_id 和 parent_span_id 时，需要通过 OpenTelemetry 的 NonRecordingSpan + SpanContext 重建父上下文。使用 contextvars 标记 reconstructed thread 以避免重复写回 trace_ctx。"

* **Cursor 输出：** 核心逻辑正确，但首次输出把 `is_remote` 设为 True（应该是 False，因为是同进程内跨线程）。纠正后代码可用。

### 1. 任务背景

* 在 Parallel Flow 中，子任务在线程池中执行时 OTEL Context 丢失，导致 Langfuse 上看到断裂的 trace 树。
* 需要在 `start_span` 中检测并重建父 span context。

### 2. 使用 Cursor 的过程

* **Prompt：**
  > "lazyllm 的 Parallel Flow 在线程池执行子任务。globals 中的 trace_ctx 已正确传播了 trace_id 和 parent_span_id，但 OpenTelemetry 的 Context 是 thread-local 的。请在 start_span 方法中，当检测到 is_root_span 为 True 但 trace_ctx 中有 trace_id 和 parent_span_id 时，通过 NonRecordingSpan(SpanContext(...)) 重建 parent_context 并传给 start_as_current_span。同时用 contextvars 标记当前线程为 reconstructed，避免覆写 trace_ctx。"

* **Cursor 第一次输出：** 整体思路正确，但 `is_remote=True`，且缺少 `TraceFlags(0x01)` 导致 span 不会被采样。
* **修正 Prompt：** "is_remote 应该为 False（同进程内跨线程），TraceFlags 需要设为 SAMPLED(0x01)"
* **最终效果：** 修正后代码可用，已在本地验证 Parallel Flow 的 span 树完整性。

### 3. 人机分工

* Cursor：生成 SpanContext 重建代码框架、contextvars 标记逻辑
* 人工：判断 is_remote 语义、确认 TraceFlags 值、设计条件判断逻辑（何时重建 vs 正常创建）

### 4. 结果评估

* **打分：6/10**
* **最大价值点：** 省去了查 OpenTelemetry API 文档的时间，快速拿到可用代码框架
* **最大问题：** 在分布式追踪语义细节上不可靠，is_remote / TraceFlags 等关键参数出错

# 二、典型失败案例

## 1. 失败任务背景

* **目标：** 在 `lazyllm/tracing/__init__.py` 中实现延迟导入机制（`_SUBMOD_MAP` + `__getattr__`），避免 import lazyllm.tracing 时触发 OpenTelemetry 等重量级依赖的加载。

## 2. ❌ 失败 Prompt（原始）

> "请为 lazyllm/tracing/__init__.py 实现 lazy import，只在真正用到某个符号时才 import 对应子模块。"

## 3. ❌ 问题分析

* **Cursor 输出问题：**
  - 生成了标准的 `__getattr__` + `_SUBMOD_MAP` 结构，但将所有子模块的所有公开符号都放在 map 中
  - 没有区分"必须立即导入的"（如 configs 注册）和"可以延迟的"（如 runtime）
  - 导致 config 注册代码未执行，运行时 config key 缺失报错
  - 且与 `__init__.py` 中已有的 `__all__` 导出冲突

* **问题根因：**
  - **缺少上下文：** Cursor 不知道 `configs.py` 中有 `add_config` 调用必须在 import 时执行
  - **Prompt 太模糊：** 只说了"lazy import"，没有指定哪些必须 eager、哪些可以 lazy
  - **超出 Cursor 能力：** Python 的 `__getattr__` lazy import 模式与包初始化副作用之间的交互比较微妙

## 4. ✅ 改进 Prompt（优化后）

> "在 lazyllm/tracing/__init__.py 中实现部分延迟导入：
> 1. configs.py 必须在 import 时立即执行（因为它调用 add_config 注册配置项）
> 2. runtime.py 和 hook.py 可以延迟导入
> 3. backends/ 可以延迟导入
> 4. 使用 _SUBMOD_MAP = {'start_span': '.runtime', 'finish_span': '.runtime', ...} + __getattr__ 模式
> 5. __all__ 必须包含所有公开符号
> 6. 保留 from .configs import ... 的直接导入"

## 5. 🔄 改进思路

* **增加了哪些约束：** 明确区分 eager vs lazy 模块，列出每个模块的导入策略
* **是否分步骤：** 是，先生成 _SUBMOD_MAP，再生成 __getattr__，最后调整 __all__
* **是否提供示例：** 提供了 _SUBMOD_MAP 的格式示例
* **为什么生效：** 将"如何做"的决策从 Cursor 转移到了人，Cursor 只负责按规则填充代码

## 6. 🎯 最终效果

* 改进后一次通过，import 时只加载 configs，其余按需加载
* 和失败版本相比：避免了 config 未注册的运行时错误，import 时间从 ~200ms 降到 ~30ms

# 三、可复用 Prompt 模板

## 模板 1（推荐）—— 分层架构代码生成

```plain text
我需要在 {项目名} 中实现 {功能名}。

架构要求：
1. {层1名称}：{职责描述}，文件路径 {path}
2. {层2名称}：{职责描述}，文件路径 {path}
3. {层3名称}：{职责描述}，文件路径 {path}

约束条件：
- {约束1，如：延迟初始化、线程安全等}
- {约束2，如：不能直接依赖某模块}
- {约束3，如：必须兼容某接口}

请先生成 {层1} 的代码，我会逐层确认后再继续。
```

**适用场景：** 多文件、多层次的功能模块开发（如 tracing 系统、plugin 系统）

**为什么有效：** 将大任务拆为层次分明的小任务，每层有明确的边界和约束，Cursor 不会跑偏。逐层确认避免了后期大规模返工。

## 模板 2 —— Bug 修复 / 行为纠偏

```plain text
当前代码：
{贴出相关代码片段}

问题现象：{描述 actual behavior}
期望行为：{描述 expected behavior}

根因分析：{你的判断，如 "is_remote 应为 False 因为是同进程跨线程"}

请基于以上分析修改代码，只输出需要改动的部分。
```

**适用场景：** 已定位到问题原因，需要 Cursor 帮忙生成修复代码

**为什么有效：** 提供了根因分析，Cursor 不需要自行推断问题原因，直接按照给定方向生成修复代码，准确率更高。

# 四、Prompt 使用技巧

* **分步生成优于一步到位：** 本周最大的经验就是将"生成整个 tracing 模块"拆分为 runtime → backends → hook → module 集成，每步验证后再推进。一次性让 Cursor 生成 1000+ 行跨多个文件的代码，质量极不稳定。

* **提供已有代码作为上下文比描述接口更有效：** 直接 @ 引用 `globals.py`、`configs.py`、`hook.py` 等文件，比在 Prompt 中描述"globals 是一个线程安全的上下文对象"效果好得多。Cursor 能从代码中推断出更多细节。

* **对 SDK/框架的细节参数不要信任 Cursor：** OpenTelemetry 的 `is_remote`、`TraceFlags`、Langfuse 的属性命名等细节，Cursor 经常给出"看起来对但实际错"的值。这类参数必须人工查文档确认。

* **"只输出改动部分"能显著减少噪声：** 特别是在修改已有大文件时，要求 Cursor "只输出需要新增/修改的代码段"比"输出完整文件"高效得多。

* **必须拆步骤的场景：**
  - 涉及循环依赖/导入顺序的改动
  - 跨 3 个以上文件的联动修改
  - 需要同时改接口定义和所有调用方的重构

# 七、思考

* **Cursor 最适合做什么：**
  - 有明确规格的中等规模代码生成（50-200 行/次）
  - 重复模式代码（如多个 backend 的 attribute mapping）
  - 已有代码的小规模重构（如 import 路径调整、变量重命名）
  - 根据已有代码推断并补充类似逻辑

* **Cursor 最不适合做什么：**
  - 涉及 SDK 底层语义细节的判断（如 OpenTelemetry is_remote 参数）
  - 跨模块循环依赖的诊断和解决
  - 需要理解运行时行为（线程模型、context 传播）的架构决策

* **给团队的 3 条建议：**
  1. **永远先拆分再生成。** 将大任务拆为 3-5 个有明确边界的子任务，每个子任务独立让 Cursor 生成。一次性生成大量代码的质量不可控。
  2. **把"你知道的"写进 Prompt。** 如果你已经知道根因、知道该用什么 API、知道哪些约束条件，全部写进去。让 Cursor 做"执行者"而非"决策者"时效果最好。
  3. **对第三方 SDK 参数做人工交叉验证。** Cursor 在标准库和主流框架上表现好，但对细分 SDK（OpenTelemetry、Langfuse）的参数语义经常有微妙错误，必须查文档确认。
