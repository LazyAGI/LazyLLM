# LazyLLM Git PR Review：多角色协作工程复盘（修正版）

> **角色前提（必读）**  
> - **Cursor**：承担 `lazyllm.tools.git.review` 整套 review 逻辑的设计与实现；并用这套逻辑去 **review 目标 PR（他人/其他 Agent 写的业务代码）**。  
> - **Codex**：不维护 review 系统本身，而是扮演 **「其他开发者」**——根据 review 意见去 **修复被 review 的那份代码（目标 PR）**。  
> - **ClaudeCode**：不维护 review 系统本身，而是扮演 **「评委」**——对 **目标 PR 上的修复是否合理**、或 **review 意见是否成立** 做二次判定（具体以你在该轮对话里给的输入为准）。  
> - 因此：**很多工程阶段只有 Cursor 在迭代 review 管线**；Codex / ClaudeCode **并非全程在场**，只在「目标 PR 需要修」或「需要裁判」时介入。

---

## 1. 背景与初始状态

- **目标**：把 LazyLLM 的 Git PR Review 从「单次调用 / 弱上下文」演进为 **多轮、可观测、可缓存、可恢复、可提交 GitHub** 的工程化管线，并能对真实 PR 给出 **架构级** 与 **实现级** 两类价值。  
- **初始 Cursor 方案**：以 `review.py` 为起点，逐步拆分为 `lazyllm/tools/git/review/` 包；引入预处理（arch / 历史规范）、R1–R5（命名随迭代变化）、checkpoint、poster、预算与策略等。  
- **多角色是否一开始就存在**：**否**。管线主要由 Cursor 迭代完成；Codex / ClaudeCode 是在 **「用 review 去审别人写的 PR」** 这条真实闭环里逐步引入的 **外部角色**，不是从第一天就固定进开发流程的「三位一体」。

---

## 2. 整体协作架构（修正版）

### 2.1 两条链路要分开看

**A. 管线研发链路（多数时候只有 Cursor）**

```
需求/缺陷反馈 → Cursor 设计/改代码 → Cursor 自测（日志、样例 PR、本地跑通）
```

**B. 目标 PR 闭环链路（Cursor + 外部角色）**

```
目标 PR（他人/Codex 写的业务改动）
    → Cursor 用 review 管线产出意见（含多轮分析与去重）
    → Codex 按意见改目标 PR（开发者）
    → ClaudeCode 评判修复/意见是否合理（评委）
    → 必要时回到 Cursor 调整 review 策略/Prompt/预算（改的是「评审器」，不是替 Codex 写业务）
```

### 2.2 角色定位（按你的定义）

| 角色 | 职责 | 典型产出 |
|------|------|----------|
| **Cursor** | Review 系统的作者；对目标 PR 执行 review | `rounds.py`、`pre_analysis.py`、文档、策略与预算 |
| **Codex** | 「其他开发者」：改 **被 review 的仓库/PR** | 业务代码修复、补测试、改文档（非 review 包本体） |
| **ClaudeCode** | 「评委」：对 **修复质量** 或 **评审是否靠谱** 给独立意见 | 结构化评判、风险点、是否接受该修复 |

### 2.3 这个架构解决什么问题？

- **把「评审器研发」和「被评审的业务改动」解耦**：review 包可以独立演进，不依赖 Codex 每天改它。  
- **降低「自嗨式评审」**：目标 PR 由他人实现，Cursor 的 review 更容易暴露「提示词/上下文/策略」的真实短板。  
- **引入独立裁判**：ClaudeCode 作为评委，缓解「同一套模型既写评审器又评自己」的心理偏差（注意：它评的仍是 **目标 PR 侧**，不是替 Cursor 给 review 包做 code owner）。

## 3. 过程分阶段总结（合并多会话，非流水账）

> **说明**：下列阶段把跨多轮对话的诉求合并成「演进主线」。**不逐条对应某一次聊天**，但保留对复盘有价值的技术事实（管线能力、典型故障、策略取舍）。**Codex / ClaudeCode** 主要在「目标 PR 修与评」闭环中出现；**review 包本体的设计与实现** 以 **Cursor** 为主。

### 阶段 1：从单文件到可运行多轮管线（Cursor）

- **阶段目标**：由单次调用演进为 **多轮 pipeline**，输出 **结构化**、可对接 GitHub；具备 **可观测、可恢复** 的工程属性。  
- **落地内容（合并多会话）**：  
  - 将 `review.py` **拆包** 为 `lazyllm/tools/git/review/`，按职责分文件（runner、rounds、pre_analysis、checkpoint、poster、constants 等），避免单文件无限膨胀。  
  - 建立 **R1→R2→…→终局合并/发帖** 的主干（轮次命名随迭代有 R4a/R4/R5 等拆分，以代码与 `git_review_pipeline.md` 为准）。  
  - **预处理**：仓库克隆、架构/历史规范等 **pre_analysis**，为后续轮次提供共享上下文。  
  - **工程化**：进度展示（阶段与子任务粒度）、关键步骤 **重试/退避**（如 QPS）、**checkpoint** 持久化中间结果；**pre_round** 类 PR 级摘要，供多轮复用；**批量提交** review（必要时补全 Git 侧接口能力）。  
- **你的关键输入**：对标 Cursor/OpenCode——**R2 由 agent 自主拉上下文**（读文件、grep 等），而不是只看 hunk 邻域。  
- **典型问题（模式）**：管线刚跑通时，问题集中在 **实现正确性**（路径、接口、是否真写入缓存），而非「评得准不准」。  
- **与 Codex/ClaudeCode**：本阶段几乎全是 **评审器研发**；外部角色尚未成为主线。

### 阶段 2：R1/R2 分工、上下文与「评论管线」（Cursor）

- **阶段目标**：让 **局部 diff 评审** 与 **关联代码探索** 形成清晰分工，并开始处理 **Prompt 体积** 与 **去重输入** 的可控性。  
- **落地内容（合并多会话）**：  
  - **R1**：以 **hunk + 当前文件窗口** 为主（大文件按行数截断上下文等策略），偏「快扫 + 定位」。  
  - **R2**：**React/agent** 路径，用工具扩展上下文（调用链、相关模块）；并讨论 **R2 如何携带、修正 R1 意见**（而非仅在最后去重时才见面）。  
  - **R4/R5 侧**：拉取 **PR 已有评论** 参与去重，避免与人工重复；**评论压缩**（摘要字段、路径/行号/ID 等结构化输入）支撑后续 LLM 去重。  
  - **阶段 2 失败与上下文上限**：出现过「阶段 2 因上下文过长失败」的专项，对策包括：**简化 Prompt**、**分文件再汇总**、**不把整份 R1 全量灌入**、对单轮上下文设 **硬上限（如 4K 量级）** 与压缩策略——这些属于 **评审器侧** 的明确工程记忆，不应在合并阶段被抹平。  
- **典型问题（模式）**：**上下文不足 → 误判**（例如未看到初始化语义、调用关系）；与 **上下文过大 → 失败/超时** 同时存在，需要 **分块、摘要、结构化传递** 而不是单一路径堆长文本。  
- **与 Codex/ClaudeCode**：当你用真实目标 PR 验收时，**误判**会暴露为「业务侧觉得 review 不靠谱」；**Codex** 修目标分支、**ClaudeCode** 评修复是否成立——但根因仍在 **评审器上下文与策略**，需回到 Cursor 迭代。

### 阶段 3：R2/React 稳定性、失败可见性与外部 I/O（Cursor）

- **阶段目标**：解决 **agent 不收敛**、**失败静默**、**clone/API/网络** 等「跑不完、难定位」问题，使管线可长期维护。  
- **落地内容（合并多会话）**：  
  - **工具循环不产出终局 JSON**：讨论并落地 **max 步数 + 强制总结**（如 force summarize 类机制），避免无限工具调用；同步涉及 **ReactAgent** 行为与文档。  
  - **路径与数据就绪**：如 **clone/workdir 未就绪** 导致 R2 跳过、缓存写到 **非预期目录**（如 `/tmp` vs `config['home']`）等——属于典型 **集成级 bug**，合并会话时 worth 保留为「检查清单项」。  
  - **GitHub**：**401** 引导登录、**分页/列表上限**（如历史 PR 数量与 `max_history_prs` 预期不一致）、**代理/连接错误** 时的跳过或重试策略；发帖侧 **422 行号不可解析**、批量失败 **fallback 单条** 等。  
  - **进程语义**：你明确要求 **agent/关键步骤失败应失败退出或可感知**，而非静默继续——这是 **流程设计** 层面的结论，应写在复盘里而非只写「不稳定」三个字。  
- **典型问题（模式）**：**环境/网络** 与 **代码逻辑** 交织时，若日志不足会误判为「模型不行」；实际上大量是 **I/O 契约与容错** 问题。  
- **与 Codex/ClaudeCode**：本阶段仍主要是 **评审器** 可靠性；评委适合用来判断「跳过/降级策略是否损害大 PR」，但实现仍在 Cursor。

### 阶段 4：缓存分层、断点续跑与预分析资产（Cursor）

- **阶段目标**：区分 **全局可复用资产** 与 **单次 PR 临时态**；支持 **从某一阶段恢复** 而不重复昂贵前置步骤。  
- **落地内容（合并多会话）**：  
  - **两级缓存**：例如 **仓库级**（架构、历史 review 规范等）与 **PR 级**（各轮中间结果、clone 工作副本）；路径从「带 repo 名的散落命名」演进为 **`config['home']/review/cache/<repo>/...`** 等更可维护约定（以当前实现为准）。  
  - **阶段枚举**：`ReviewStage`（或等价机制）表达 **从哪一轮恢复**；区分「**逻辑未变可复用**」与「**缺缓存则 warning 并重算**」等行为。  
  - **资产保留**：如 **arch / spec** 类文件 **不应在任务结束后被误删**；日志曾出现「提示已保存但实际未落盘」类问题，属于 **复盘应记一笔** 的故障类型（便于以后做回归）。  
  - **clone 行为**：远程更新时 **pull**、代理是否作用于子进程等 **环境说明** 曾反复讨论，适合作为「运维注意事项」保留摘要。  
  - **历史 PR 规范**：按 PR **分别总结再合并**、展示 **当前处理到第几个 PR** 等体验诉求，属于 **预分析阶段产品化** 的一部分。  
- **典型问题（模式）**：缓存 **键设计** 与 **失效条件**（如 `max_history_prs` 变大时的增量重算）若不清，会出现「以为命中缓存其实悄悄过期」。  
- **与 Codex/ClaudeCode**：弱相关；若目标 PR 长跑失败，恢复执行主要依赖 **评审器状态机**。

### 阶段 5：「评得准」——架构视角、设计文档与规则化约束（Cursor）

- **阶段目标**：从「找对局部问题」升级到 **模块边界、API、复用、体验一致性**；用 **设计文档** 做中介，减少纯 diff 臆测。  
- **落地内容（合并多会话）**：  
  - **架构师视角 round**：补充高层 checklist（与「只看正确性」区分）；实践中出现过 **轮次编号**（如新增能力放在 R4a/R4/R5 哪一段）与 **旧 checkpoint 兼容** 的讨论。  
  - **规则化约束**：强调 **只评论与 diff 强相关** 的改动、**历史上下文仅作证据**；**lint 类**（如 unused import、复杂度）**统一不评**，避免噪声淹没架构问题。  
  - **Round 4 增强方向（示例）**：**先根据 diff + arch 生成 PR 设计文档**，再 **评审该文档**，再 **回看实现是否一致**；并行探索 **公共 API/能力清单**（常见矛盾：哪些由 **LLM 判断**、哪些由 **规则/路径** 约束以抑幻觉）。  
  - **AGENTS.md / agent skill**：讨论是否 **优先 ingest** 仓库已有约定，以及在无文件时是否在 arch 中 **生成等价约束**，减少重复踩坑。  
  - **静态分析**：从偏 Python 扩展到 **多语言** 的规划与落地（与「lint 不评」策略要分清：工具可跑，但 **评审输出要克制**）。  
  - **R2 效率**：合并 hunks、共享预热上下文（如 shared context）、**工具与批读**、**输出侧减负** 等，目标是 **少调用、每次更饱满**——这是独立的一簇优化，不应在合并阶段只剩一句「优化 R2」。  
- **典型问题（模式）**：**Prompt 写得像架构师，但产出像 linter** → 需要 **负例 + 硬约束 + 分层模板**；**公共能力清单幻觉** → **规则子集 + 路径证据** 与 LLM 分工。  
- **与 Codex/ClaudeCode**：目标 PR 若按设计大改，**Codex** 动业务、**ClaudeCode** 评「设计落地是否一致」特别匹配；评审器侧仍是 Cursor。

### 阶段 6：全局预算、策略可解释与文档/指标闭环（Cursor）

- **阶段目标**：显式处理 **覆盖 vs 成本 vs 时延**；用 **文档** 固定「当下真相」，用 **指标** 支撑迭代，而不是凭感觉。  
- **落地内容（合并多会话）**：  
  - **BudgetManager / 自适应策略**：token、调用次数等 **预算**；R2 的 **group/chunk** 等模式切换；曾讨论 **大 PR 是否应跳过关键阶段**——这类策略必须用 **业务后果** 检验（往往「越大越不该跳过」）。  
  - **代码体量**：在不大改语义前提下做 **精简与复用**（行数目标、函数合并等），与 **可靠性** 之间存在张力，需要 **回归 PR 集** 约束。  
  - **文档**：`git_review_pipeline.md`、中文 Best Practice（流程、每轮大致调用与 token 量级、缓存语义）与 **Evaluation & Metrics** 章节诉求——让团队能回答「覆盖了吗、质量如何、贵不贵」。  
- **典型问题（模式）**：**缺指标** 会导致优化方向漂移（只省钱或只加轮次）；**策略误伤大 PR** 属于「看起来省算力、实际上毁价值」的典型坑。  
- **与 Codex/ClaudeCode**：指标里若包含「人类/Codex 对意见的采纳与返工率」，才和外部角色强相关；评审器研发仍以 Cursor 为主。

## 4. 关键转折点（Highlight）

1. **真实目标 PR 进入流程**：把问题从「实现是否正确」转成「评审是否可靠」，这是质量跃迁最常见的触发器。  
2. **R2/Agent 可控性**：强制总结、步数上限、工具与提示词约束，通常比单纯换模型更能稳定闭环。  
3. **评审约束明确化**：仅评 diff 相关、压制 lint 噪声、强调证据链——往往比再加一轮模型更有效。  
4. **文档与指标并行**：没有 metrics，团队无法判断这轮改的是「成本」还是「质量」。

## 5. 问题类型归纳（标注：易发于谁 / 适合谁处理）

| 问题类型 | 易发于 | 适合处理 | 如何避免（可执行） |
|----------|--------|----------|--------------------|
| 上下文不足导致误判 | Review 管线（Cursor 实现） | 调上下文拼装/缓存/分块策略 | 定义最小充分上下文；对长文件分段；关键符号定位 |
| Prompt 退化为「lint/局部挑刺」 | 评审 Prompt/策略 | Cursor 改规则与负例 | 明确禁止项；要求引用证据；架构项单独模板 |
| Agent 循环不收敛 | R2/React 路径 | Cursor 改流程 | max steps、强制总结、工具白名单、输出 JSON 契约 |
| 评审意见与业务修复脱节 | 目标 PR 修改者（Codex） | Codex 修代码 + ClaudeCode 评修复 | review 意见写清「期望行为/边界条件/验收点」 |
| 外部服务不稳定 | 环境/网络/GitHub | Cursor 做容错与可恢复 | 重试、降级、跳过、缓存；错误可诊断日志 |
| 「策略看起来合理但业务上错误」 | 策略/阈值 | ClaudeCode 作独立评委 + 人类最终裁决 | 大 PR 不 skip 关键阶段；阈值与理由写进文档 |

## 6. 多角色模式总结（修正版）

- **为什么有效**：评审器研发（Cursor）与被评审业务（他人 PR）分离，更容易暴露评审器短板；评委（ClaudeCode）提供第二视角，降低「评审意见本身不靠谱」带来的返工。  
- **单角色 vs 多角色**：单 Cursor 也能把管线写完整；多角色的价值主要体现在 **目标 PR 闭环**（有人真改、有人真评），而不是「每个阶段都必须三人同时在场」。  
- **何时需要 ClaudeCode（评委）**：  
  - 评审意见「看起来对但实现会翻车」  
  - Codex 修复引入副作用  
  - 需要独立视角做 **接受/拒绝** 与 **风险分级**  
- **冗余与优化**：不要在每个小改动都拉评委；把 ClaudeCode 用在 **高冲突/高风险/大 diff** 的节点。

## 7. 可复用方法论（Checklist）

### 使用 Cursor 研发 review 管线

- 每个 round 写清 **输入/输出契约** 与 **失败可观测性**（日志定位到子阶段）。  
- 先保证 **确定性步骤**（解析 diff、定位行号、缓存键）正确，再堆 LLM 智能。  
- 用 **真实 PR 回归集** 驱动迭代，而不是只跑自造样例。

### 如何引入 Codex（开发者）

- 把 review 意见整理成 **可执行任务**：现象、根因假设、期望行为、影响面、验收方式。  
- 明确 **改动范围**（只动目标 PR 分支业务代码），避免 Codex 误改评审器仓库。  
- 修复后保留 **最小复现** 与 **对比 diff**，方便评委与人类审阅。

### 如何用 ClaudeCode（评委）

- 输入建议包含：**目标 diff**、**相关上下文摘录**、**开发者修复说明**、**需要裁决的问题清单**。  
- 输出要求 **结论先行**（接受/有条件接受/拒绝）+ **理由** + **残留风险**。  
- 仅在 **高成本节点** 启用，避免评委疲劳与标准漂移。

### 多轮迭代如何收敛

- 每轮迭代必须绑定 **指标**（耗时、token、覆盖率、误报率样本）。  
- 「改 Prompt」与「改流程」二选一时：先定位是 **证据不足** 还是 **约束不足**。  
- 评审器变更要做 **回归 PR 集**，避免优化 R2 破坏 R4/R5。

### 设计稳定的「自动 review + 人类/开发者闭环」

- review 侧：契约化、可缓存、可恢复、可降级。  
- 业务侧：固定「开发者-评委」接口（任务单格式、验收标准）。  
- 组织侧：明确 **谁改业务、谁评业务、谁维护评审器**，三者不要混岗到同一语境里自欺欺人。
